# Session 7 - Wireless: WiFi AP and Web Dashboard

The pet gets a web dashboard. It starts a WiFi access point called "VirtualPet", serves an HTML page with CSS and JavaScript, and pushes live stat updates over a WebSocket every 500ms. You can feed, play, put to sleep, bathe, heal, give water, save, rename, and reset your pet from any browser on the WiFi network.

## What we did

WirelessManager handles everything: the WiFi AP, the HTTP server, and the WebSocket server. The ESP32's built-in WiFi hardware handles the AP mode - no external module needed.

### WiFi Access Point

begin() calls WiFi.softAP("VirtualPet") which creates a WiFi network that phones and laptops can connect to. No password. The ESP32 acts as a router, assigning IP addresses via DHCP. The device's own IP is always 192.168.4.1 in AP mode. Open a browser to that address and the dashboard loads.

### HTTP Server

A WebServer instance on port 80 handles three routes:
- / serves the HTML page
- /style.css serves the CSS
- /script.js serves the JavaScript

The page is stored in PROGMEM as a constant string (dashboard_assets.h, generated by a Perl script from the .html/.css/.js files). The HTML has a {{PET_NAME}} placeholder that gets replaced with the actual pet name at serve time.

The HTML is a full dashboard page with stat rows, a mood badge, action buttons (Feed, Play, Sleep, Bathe, Heal, Drink, Save), a rename input field, and a death/respawn overlay. The CSS styles it with card layouts, coloured stat bars that match the device's bar colours, and an overall dark theme.

### WebSocket Server

A WebSocketsServer on port 81 provides live updates. Every 500ms, broadcastStats() builds a JSON payload with all stats and sends it to every connected client. The JSON looks like:

```json
{
  "pet": {
    "name": "Pixel",
    "mood": "happy",
    "alive": true
  },
  "stats": {
    "fullness": 72,
    "happy": 65,
    "energy": 80,
    "cleanliness": 55,
    "sick": 10,
    "hydration": 74,
    "tired": 25,
    "sad": 5
  }
}
```

The JavaScript client connects to the WebSocket on page load, receives these updates, and patches the DOM in place - no page refresh needed. The stat bars animate smoothly, the mood badge updates, and the death overlay appears when alive switches to false.

The JS also sends commands back to the device through the WebSocket. Pressing a dashboard button sends {"action":"feed"}, {"action":"play"}, etc. The C++ side parses the JSON with simple string operations (no external JSON library - just indexOf and substring) and calls the matching Pet method.

The rename feature sends {"action":"setName","name":"Fluffy"} and the device updates the pet name in both the firmware and NVS. The reset button sends {"action":"reset"} which sets a flag that main.cpp checks in the loop.

## Things that tripped me up

- The HTML, CSS, and JS files need to be embedded into the firmware binary. They can't be on a filesystem because there's no SD card and no SPIFFS setup. The embed_file.pl script converts them into PROGMEM string constants in dashboard_assets.h. Any change to the dashboard files means regenerating this header.
- WebSocket events on the C++ side use std::bind with placeholders for the callback. The syntax is ugly but it's the standard Arduino WebSockets library pattern. The dev roadmap notes that lambda syntax is not allowed in the project codebase.
- The JSON parsing on the C++ side is deliberately simple - just indexOf and substring to find "action":"...". This works because the JSON is small and the client is trusted (it's your own device). A production system would need proper JSON parsing, but that would add a dependency for no real benefit here.
- The broadcast interval is 500ms. If you lower it too much, the WebSocket traffic can overwhelm the ESP32's CPU and interfere with the game loop. At 500ms it's smooth on the browser without affecting pet behaviour.
- The death overlay in the JavaScript fires on the WebSocket message that reports alive:false. The dashboard then hides all the controls and shows a "Your pet has died" message with a respawn button. The respawn button sends the reset command.
- Soft AP mode means the device creates its own network. It can't simultaneously connect to the internet as a client. If you need internet access while the dashboard is running, you'd need a different WiFi mode or a second radio.
